# Veil Scout 机制审查 — 补充问题

> 本文件是对 `疑问.md` 的**补充**，列出文档作者尚未注意到的机制漏洞、合约安全风险与激励缺陷。
> 按严重程度排序，每条包含：**问题说明、严重程度、可缓解层级、建议方案**。

---

## 🔴 Critical（可能直接导致系统失效或资金损失）

### C1. 赢家的积分从哪里来？零和悖论

**问题说明**：
文档说"预测正确 → 获得 Scout Credits"，但 Credits 是**不可转让、不可提现**的内部积分。如果市场是零和的（输家的 credits 转移给赢家），那么当所有人都预测正确时，赢家赚谁的 credits？

更严重的是：如果系统**额外 mint credits** 给赢家，那么总 credits 量会通胀，赛季末排行榜分数失去可比性（前期参与者分数被稀释）。

**严重程度**：🔴 Critical — 经济模型根基问题，不解决则整个积分系统逻辑不自洽。

**可缓解层级**：产品层 + 合约层

**建议方案**：
1. **明确积分来源**：Settlement 时，赢家获得的 credits = 自己的本金 + 输家的本金 × 分配系数。无额外 mint。
2. **引入"手续费"抽成**：每笔交易抽 1-2% 手续费进入赛季池，赢家分配时含手续费再分配 → 系统长期有 credits 流入。
3. **如果必须通胀奖励**：设计赛季内固定的"奖励预算 credits"，赛季初一次性 mint 到合约，结算时从预算池分配。确保总供应量可知。

```solidity
// 建议：Settlement.sol 中
totalLosingCredits = sum(输家投入);
winningPool = totalLosingCredits * (100 - protocolFee) / 100;
// 赢家按持仓比例分配 winningPool
```

### C2. 项目方内幕交易（Rug Pull 市场）

**问题说明**：
项目方提交自己的 milestone（如"30 天内完成 1,000 笔支付"），然后自己是最清楚能否达成的人。项目方可以：
- 如果知道自己做不成 → 提前重仓买 NO
- 如果知道自己能做成 → 提前重仓买 YES
- 甚至故意操纵结果（最后一刻不完成 milestone）来确保自己赢

文档完全没有限制"项目方自己参与预测"。

**严重程度**：🔴 Critical — 内幕信息优势摧毁市场公平性。

**可缓解层级**：产品层 + 合约层（部分）

**建议方案**：
1. **合约层**：`MarketFactory.createMarket()` 记录 `projectOwner` 地址，该地址及其关联地址禁止在该市场交易（通过链下监控 + 前端屏蔽，合约层可做 `blocked[address]` 映射）。
2. **产品层**：项目方提交 milestone 时需质押保证金，若被证实恶意操纵，保证金罚没。
3. **更现实的方案**：AI 分析师在生成报告时，标记"该项目方是否有交易历史"，前端对项目方地址做风险提示。

### C3. 单边持仓规则形同虚设

**问题说明**：
文档说"每个 Scout 在每个市场只能持有单边仓位（YES 或 NO，不能两个都持有）"。

但合约只能约束**地址级别**（`mapping(address => Position)`），无法约束**人级别**。一个控制 10 个钱包的人，可以用 5 个钱包买 YES、5 个钱包买 NO，实现完美对冲。zkID 只防"重复领取 credits"，不防"用 credits 后多地址对冲"。

**严重程度**：🔴 Critical — 防女巫的核心目的（防止一个人刷排行榜）被绕过。

**可缓解层级**：产品层（主要）+ 合约层（辅助）

**建议方案**：
1. **合约层辅助**：强制 `buy()` 时要求提交 `zkProof`，证明该 zkID 本赛季未在该市场反向持仓。即：zkID 级别限制单边，而非地址级别。
   ```solidity
   // 需要 zkID 与地址绑定
   mapping(bytes32 nullifierHash => mapping(uint256 marketId => Side)) public zkPosition;
   ```
2. **产品层**：引入"仓位集中度惩罚"——如果一个用户在多个地址的持仓高度相关（链下分析），降低其 Scout Score。
3. **现实妥协**：MVP 阶段承认无法完全防止对冲，改为用"单市场积分上限（20%）" + "过度集中惩罚"降低套利收益。

## 🟠 High（严重设计缺陷或较大攻击面）

### H1. LMSR 在 Solidity 中的整数溢出与精度灾难

**问题说明**：
技术路线图推荐 LMSR：

$$C(q) = b \ln \left( e^{q_{yes}/b} + e^{q_{no}/b} \right)$$

Solidity **没有浮点数**。`exp(x)` 和 `ln(x)` 需要用定点数库（如 PRBMath）实现，但：
- `exp(q/b)` 当 `q` 较大时（如 10,000 credits）会**指数级爆炸**，超出 `uint256` 范围
- 即使使用 PRBMath 的 `exp`（基于 `e^x` 的泰勒展开），大输入下也会溢出或精度崩溃
- LMSR 的 gas 消耗远高于简单常数乘积 AMM

**严重程度**：🟠 High — 数学上正确的东西在 EVM 上可能根本无法实现。

**可缓解层级**：合约层

**建议方案**：
1. **MVP 改用简化定价**：固定赔率池（如 50/50 起步，每买 1 YES 涨 0.01，每买 1 NO 降 0.01），硬编码价格边界 `[0.05, 0.95]`。
2. **如果必须用 LMSR**：使用 ABDKMath64x64（64.64 定点数），但限制单笔交易大小，且 `b` 参数必须足够大（如 `b = 10000e18`）以平滑曲线。
3. **更聪明的方案**：链下计算价格，链上只验证。前端用 JS 计算 LMSR 价格，用户签名交易时附带"预期价格"，合约检查价格偏差在容差内。

```solidity
// MVP 简化版：线性价格曲线
function getPrice(uint256 yesBought, uint256 noBought) pure returns (uint256 yesPrice) {
    uint256 total = yesBought + noBought + 1; // +1 防除零
    yesPrice = 5000 + (yesBought * 5000 / total) - (noBought * 5000 / total);
    if (yesPrice < 500) yesPrice = 500;   // 最小 5%
    if (yesPrice > 9500) yesPrice = 9500; // 最大 95%
    // 返回基点（1 = 0.01%）
}
```

### H2. Leaderboard 链上排序的 Gas 爆炸

**问题说明**：
`Leaderboard.sol` 中的 `getRanking(uint256 seasonId, uint256 topN)` 如果完全在链上实现，意味着每次结算后都要维护一个排序数组。当 Scout 数量达到 100+ 时，插入排序的 gas 会高到无法使用。

技术路线图提到"或链下索引器"，但没有明确说明。

**严重程度**：🟠 High — 如果坚持链上排序，产品无法在 100+ 用户时运行。

**可缓解层级**：合约层 + 架构层

**建议方案**：
1. **链下索引器（推荐）**：合约只 `emit ScoreUpdated(address, seasonId, newScore)`，前端/TheGraph 订阅事件并排序。`Leaderboard.sol` 只存储个人分数，不做全局排序。
2. **如果必须链上**：用"分桶排名"（如 1000 分一桶，只统计每个桶的人数），获取 topN 时再遍历有限桶数。

```solidity
// 推荐：合约只做记录，不做排序
mapping(uint256 seasonId => mapping(address => int256)) public scores;
event ScoreUpdated(address indexed user, uint256 indexed seasonId, int256 newScore);
// 排序由 TheGraph / 前端完成
```

### H3. 不可转让 Credits = 无退出机制 = 用户无法止损

**问题说明**：
用户用 credits 买了 YES 后，如果发现自己判断错误（或获得新信息），**无法卖出**。传统预测市场有二级市场（卖出持仓），Veil Scout 没有。

这导致：
- 用户一旦入场就被锁定到结算日
- 降低参与意愿（因为决策不可逆）
- 市场流动性极差（只有买入，没有卖出）

**严重程度**：🟠 High — 产品体验致命缺陷。

**可缓解层级**：产品层 + 合约层

**建议方案**：
1. **允许卖出（推荐）**：修改 `Market.sol`，允许用户以当前 LMSR 价格卖出持仓（系统回购）。LMSR 天然支持双向交易（成本函数可增可减）。
2. **如果 credits 必须不可转让**：设计"平仓"机制——用户可以将自己的 YES 持仓按当前价格"卖回给系统"，系统回收 credits（不是转账给其他用户）。

```solidity
function sell(Side side, uint256 amount) external {
    // 计算卖出后可返还的 credits
    // 系统回收 credits，用户获得返还
    // credits 未发生地址间转账，满足"不可转让"
}
```

### H4. AI 初始赔率被项目方包装操纵

**问题说明**：
AI 分析师读取"GitHub 活动、合约部署、Demo 链接"来生成初始概率。项目方完全可以：
- 刷 GitHub commit（无意义提交）
- 部署一个空合约（无实际功能）
- 购买虚假链上交易（自己给自己转账制造"活跃"假象）

AI 被欺骗后给出虚高初始概率（如 YES = 0.85），人类 Scout 看到后会跟风买 YES，最终项目方获利。

**严重程度**：🟠 High — 信息源污染导致整个定价机制失真。

**可缓解层级**：产品层（AI 层）+ 合约层（间接）

**建议方案**：
1. **AI 层**：引入"质量权重"而非"数量权重"。如 GitHub 权重看 `merged PR 数 / 唯一贡献者数` 而非 `commit 数`。
2. **合约层**：初始赔率不直接由 AI 设定，而是有一个**社区质疑期**（如 24 小时）。质疑期内若收到足够多 evidence 反驳 AI 判断，初始赔率被调整至 0.50（中性）。
3. **数据源多样化**：链上数据通过预言机读取（难伪造），GitHub 数据作为辅助（易伪造）。

## 🟡 Medium（值得注意的设计缺陷）

### M1. 赛季过渡时未结算市场的处理

**问题说明**：
假设 Season 1 在 6 月 30 日结束，但某个市场的截止日是 7 月 15 日。文档没有说明：
- 这个市场的 credits 怎么处理？
- 用户在 Season 1 买的仓位，算 Season 1 还是 Season 2 的分数？
- 如果强制结算，对未到期项目不公平；如果顺延，赛季排名何时确定？

**严重程度**：🟡 Medium — 边界情况，但会影响赛季最终排名。

**建议方案**：
- 合约层：`createMarket()` 时要求 `deadline <= seasonEnd`，否则拒绝创建。
- 或者：`seasonEnd` 后只锁定积分领取和新市场创建，已有市场继续运行到各自 deadline，但它们的分数计入下一赛季。

### M2. zkID 匿名性与 SBT 声誉的可追踪性矛盾

**问题说明**：
用户通过 zkID 证明自己是"唯一人类"，但获得的 Alpha Scout SBT 是**公开、不可转让、永久绑定到地址**的。

长期看：
- 一个地址持有 3 个赛季的 Alpha Scout SBT
- 链上分析可以将该地址的交互模式、交易时间、gas 价格偏好与真实身份关联
- **zkID 的匿名性被 SBT 的持久性逐步侵蚀**

**严重程度**：🟡 Medium — 隐私设计的长期矛盾。

**建议方案**：
- SBT 绑定到 `nullifierHash` 的派生地址（而非用户交易地址），用户每赛季可更换前端地址但保留声誉。
- 或者：SBT 只记录"某匿名实体获得过 Alpha Scout"，不记录连续赛季关联。

---

### M3. 证据评分（Evidence Score）的可刷性

**问题说明**：
文档说"Scout C 提交 evidence → 如果验证有用，获得证据分数奖励"。但：
- 如果 evidence 质量由人工/委员会评定，评定者可能被贿赂
- 如果由 AI 评定，AI 可以被 prompt injection 欺骗
- 用户可以批量提交低质量但格式正确的 evidence（如"该项目官网打不开"这种无害但无用的信息）刷分

**严重程度**：🟡 Medium — 刷 evidence 成本低，收益确定。

**建议方案**：
- 同一用户在同一市场最多提交 N 条 evidence
- evidence 被采纳后才计分，且分数与市场结果挂钩（只有预测正确且 evidence 被采纳才给分）
- 引入"evidence 质押"：提交 evidence 需质押少量 credits，若被判定为垃圾 evidence，质押罚没

---

### M4. 合约升级性与赛季持久性

**问题说明**：
60 天 MVP 可能不会考虑升级性，但：
- 赛季制意味着合约需要跨多个赛季持续运行
- 如果 Season 1 发现 bug（如分数计算错误），如何修复而不丢失历史数据？
- 如果部署新合约，旧赛季的 SBT 和声誉如何迁移？

**严重程度**：🟡 Medium — 长期运维问题。

**建议方案**：
- MVP 阶段使用 **UUPS 代理模式**（Foundry 有 `openzeppelin-contracts-upgradeable`）
- 核心存储（scores、SBT、赛季历史）放在独立合约，业务逻辑合约可升级
- 至少保留 `Pausable` 和 ` Ownable` 用于紧急暂停

---

### M5. 前端 Mock 数据与真实 ABI 的接口漂移风险

**问题说明**：
技术路线图建议 Track C 前 2 周用 hardcoded mock 数据开发。但 Day 0 的 ABI 草案是"草案"，实际开发中 Track A 很可能调整接口（如参数顺序、事件字段、返回值类型）。

如果 Track A 第 10 天发现某个函数需要额外参数，Track C 的前端代码可能需要大规模重写。

**严重程度**：🟡 Medium — 协作效率问题。

**建议方案**：
- Track A 在 Day 0 就写 Solidity **接口文件（I*.sol）** 而非只是文档，用 Foundry 编译生成确定 ABI
- 接口一旦确定，用 `interface` 约束实现，减少后期变动
- Track A 提供 Mock 合约（部署在 Anvil），Track C 直接对 Mock 合约做真实 `wagmi` 调用，而非 hardcoded 数据

## 🟢 Low（值得关注但可以接受）

### L1. Scout Score 公式未定义，可博弈空间大

**问题说明**：
文档说 `Scout Score = 虚拟 PnL + 准确率 + 早期发现 + 证据 - 惩罚`，但没有给出具体公式。不同项的权重决定了系统的激励方向：
- 如果"早期发现"权重太高 → 用户专挑开盘时 all-in，不分析证据
- 如果"证据"权重太高 → 用户刷 evidence
- 如果"准确率"权重太高 → 用户只押确定性高的事件，回避有信息价值的争议市场

**严重程度**：🟢 Low — MVP 阶段可以手动调参，但需要尽早定义公式。

**建议方案**：
```
Scout Score = 
  0.5 × 虚拟 PnL（标准化到赛季均值）
+ 0.2 × 准确率（正确预测数 / 总预测数）
+ 0.2 × 早期发现分（按买入价与结算价的偏差加权）
+ 0.15 × 证据分（被采纳 evidence 的累计分）
- 0.1 × 集中度惩罚（单市场持仓 > 20% 时触发）
```
（权重总和 > 1 是刻意的，让表现优异者拉开差距）

### L2. 项目方提交 Milestone 的动机不足

**问题说明**：
项目方为什么要主动来 Veil Scout 提交 milestone 让人预测？
- 如果预测结果差 → 负面曝光
- 如果预测结果好 → 也只是"获得验证"

文档中没有明确项目方的正向激励（如免费流量、孵化器推荐优先权）。如果项目方不来，市场就没有标的，系统空转。

**严重程度**：🟢 Low — 产品运营问题，非技术问题。

**建议方案**：
- 项目方提交 milestone 免费，但获得平台曝光
- 预测结果好的项目进入"HTX 生态观察名单"
- 未来可收取项目方 Listing Fee（但 MVP 阶段免费）

---

### L3. AI 分析 vs 人类 Scout 的激励失衡

**问题说明**：
如果 AI 初始赔率太准，人类 Scout 没有动力参与（无 alpha 可挖）。如果 AI 太不准，市场失去锚点。

文档说"AI 是流动性提供者和话题发起者"，但没有机制确保 AI 误差率维持在合理区间。

**严重程度**：🟢 Low — 长期产品问题。

**建议方案**：
- 不公开 AI 的具体置信度，只公开概率区间（如"60%-70%"）
- 记录 AI 历史准确率，如果某类项目 AI 持续不准，提高该类项目的人类 Scout 奖励系数

## 总结：优先级矩阵

| 问题 | 严重度 | 能否在合约层解决 | 建议处理时机 |
|------|--------|-----------------|-------------|
| C1. 积分来源零和悖论 | 🔴 | 部分（手续费抽成） | Day 0 必须定稿 |
| C2. 项目方内幕交易 | 🔴 | 部分（禁止项目方地址交易） | MVP 必须实现 |
| C3. 单边持仓规则绕过 | 🔴 | 部分（zkID 级限制） | MVP 尽量实现 |
| H1. LMSR 整数溢出 | 🟠 | 是（改用简化定价） | Day 1-2 决定 |
| H2. Leaderboard Gas 爆炸 | 🟠 | 是（链下排序） | Day 1-2 决定 |
| H3. 无退出机制 | 🟠 | 是（系统回购） | 第一阶段内实现 |
| H4. AI 赔率被操纵 | 🟠 | 间接（质疑期机制） | 第二阶段 |
| M1-M5 | 🟡 | 是/部分 | 第一阶段至第二阶段 |
| L1-L3 | 🟢 | 部分/否 | 第二阶段或后期 |

---

> **给 Track A 的核心建议**：
> 
> 1. **不要用 LMSR 做 MVP** — 整数数学风险太高，先用简化线性价格曲线。
> 2. **Leaderboard 坚决链下排序** — 合约只 emit 事件。
> 3. **明确积分流向** — Settlement 必须定义清楚"赢家赚谁的钱"。
> 4. **预留系统回购接口** — 即使 MVP 不做卖出，合约里先留好 `sell()` 函数位置。
> 5. **项目方黑名单** — `MarketFactory` 创建市场时记录 `projectOwner`，合约层可扩展为禁止其交易。
